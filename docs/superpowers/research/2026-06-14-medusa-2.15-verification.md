## 2. `medusa-config.ts` registration shapes ŌĆö [High]

**Crucial distinction:** a payment/fulfillment **provider** registers *inside its parent commerce module's `options.providers` array* (with a required `id`), never as a top-level `modules` entry. A **custom module** registers as a top-level `modules` entry (`resolve` + optional `options`, no `id`, no `providers`).

```ts
// Payment provider ŌĆö nested in the Payment module
{ resolve: "@medusajs/medusa/payment",
  options: { providers: [ { resolve: "@medusa-cz/payment-comgate", id: "comgate", options: {} } ] } }

// Fulfillment provider ŌĆö nested in the Fulfillment module
{ resolve: "@medusajs/medusa/fulfillment",
  options: { providers: [ { resolve: "@medusa-cz/fulfillment-packeta", id: "packeta", options: {} } ] } }

// Custom module (e.g. Fakturoid) ŌĆö top-level, no id/providers wrapper
{ resolve: "@medusa-cz/invoicing-fakturoid", options: {} }
```
Provider entry fields: `resolve` (npm name or path), `id` (required, unique), `options` (optional). List each commerce module `resolve` only **once**; put all its providers in the single `providers` array (duplicate module entries cause loader errors).
Sources: https://docs.medusajs.com/resources/commerce-modules/payment/module-options ,
https://docs.medusajs.com/resources/commerce-modules/fulfillment/module-options ,
https://docs.medusajs.com/learn/fundamentals/modules

**[Low] / watch-item:** the `plugins: []` array shape for *consuming a published plugin* in `medusa-config.ts` was not confirmable this session. Not needed for M0 (the demo consumes workspace packages and registers providers via the verified shapes above); confirm before publishing-consumption work.

## 5. `canCalculate`/`calculatePrice` bug (#9495 / #9598) ŌĆö FIXED in 2.15.x [High]

Issue #9598 is **CLOSED**; the calculated-pricing flow is fully functional in 2.15.x (verified in source). Original bug: during v1ŌåÆv2 refactor the wiring to call a provider's `canCalculate`/`calculatePrice` was lost, so calculated options (`price_type: "calculated"`, `amount: null`) errored on `GET /store/shipping-options` instead of pricing. The cart workflow `list-shipping-options-for-cart-with-pricing` now separates flat vs calculated, calls `calculateShippingOptionsPricesStep` ŌåÆ provider `calculatePrice`, and `validateShippingOptionsForPriceCalculation` calls `canCalculate` per option. Dynamic pricing landed in v2.1.1 (Dec 2024).

**For Packeta (2.15.x): fixed, no workaround.** Set `price_type: "calculated"`, return `true` from `canCalculate`, implement `calculatePrice`. Model on the official ShipStation guide (does exactly this with a storefront loading indicator).
Fix-version precision: [Medium] (couldn't pin exact closing PR; flow confirmed present in 2.15.0ŌĆō2.15.5).
Watch (separate, newer): Issue #13062 "Cannot calculate pricing forŌĆ”" is a runtime guard for a provider returning a bad/empty price ŌĆö config/impl issue, not the #9598 regression.
